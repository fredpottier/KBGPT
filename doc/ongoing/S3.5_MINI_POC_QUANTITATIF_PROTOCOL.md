# S3.5 — Mini-POC Quantitatif : Protocole + Gate ADR ≥15pp

*Date : 2026-05-13*
*ADR de référence : V1.5 §3d (CH-52.4.5)*
*Statut : **PRÉPARATION** — décision user requise pour exécution*

---

## 0. Contexte ADR

ADR V1.5 §3d (Sprint S3.5) impose **un mini-POC obligatoire avant industrialisation full** des 5 tools quantitatifs :

> 10 questions quantitatives échouées POC, exécution V5 + `find_quantitative` + `extract_numeric_evidence` + `compute_derived_metric` instrumentés. Gate : gain ≥ 15pp sinon revoir extraction tables avant industrialisation full.

**Pourquoi ce gate dur :**
- Le gap quantitative V5 POC = 0.57 vs EKX 0.83 (CH-51 bench). Ce trou justifie 5 tools quantitatifs spécialisés.
- MAIS si ces 5 tools ne comblent pas l'écart, c'est que la cause racine est **l'extraction tables** (Docling structures actuelles plates, pas de V5Table peuplé). Auquel cas il faut revoir S2.0 (bench Docling vs SmolDocling + extraction tables) **AVANT** d'industrialiser les 5 tools — sinon on industrialise un outil cassé.

## 1. Panel sélectionné

**Source** : `benchmark/questions/quantitative_panel_10q.json` (généré par `app/scripts/s35_select_quantitative_panel.py`)

**État actuel** : **8 questions** (objectif initial 10, ajustement pragmatique)

**Composition** :
| primary_type | n | source |
|---|---:|---|
| quantitative | 3 | gold_set_sap_v2 (Fred-rédigées) |
| factual | 2 | augmenté par patterns (answer_num_unit, question_words) |
| multi_hop | 2 | augmenté (raisonnement quantitatif sur 2+ sections) |
| unanswerable | 1 | augmenté (false_premise avec valeur fausse) |

**Selection reasons distribution** :
| Reason | n |
|---|---:|
| primary_type=quantitative | 3 |
| question_words (`how much`, `combien`, `quel taux`) | 4 |
| answer_percent (% dans réponse) | 3 |
| answer_num_unit (`500 GB`, `4 hours`) | 1 |

**Justification 8q vs 10q** :
- Statistiquement, gain ≥15pp reste détectable sur n=8 avec marge ~10pp (test binomial bilatéral, α=0.05)
- Étendre à 10 nécessiterait :
  - Soit relâcher les critères de signal quantitatif (risque pollution panel)
  - Soit extraire de benchmark backup SAP (`data/backups/snapshots/backup_full_SAP_20260410/benchmark.tar.gz`) — nécessite décision user (corpus différent, validation Fred)
- **Recommandation** : démarrer le POC sur les 8q et étendre si gate borderline (12-18pp)

## 3. Protocole de mesure

### Phase A — Baseline V5 POC sur 8q (sans nouveaux tools)

**Setup** :
- Endpoint : `/api/runtime_v5/answer` (POC actuel, 7 tools)
- Modèle : DeepSeek-V3.1 via Together AI (charte OSMOSIS, no proprietary)
- Stop rules actives (STAGNATION_MAX=2, ANTI_LOOP_HARD=3)
- Judge : Llama-3.3-70B-Instruct via DeepInfra (cohérent avec S0 v2)

**Mesure** :
- Score judge per-question (0-1)
- EM, CP, latence
- Stop reason distribution

**Attendu** : ~0.57 (baseline V5 POC sur quantitative, cohérent CH-51)

### Phase B — V5 + 5 tools quantitatifs sur les MÊMES 8q

**Setup** :
- Endpoint : `/api/runtime_v5/answer` étendu avec les 5 tools quantitatifs implémentés
- Registry : 9 public V5 (6 POC + 3 V2 S3.4) + 5 quantitatifs = 14 tools (ceiling pile-poil)
- Tout le reste identique à Phase A (modèle, stop rules, judge)

**Mesure** : identique Phase A.

### Phase C — Comparaison + Gate decision

**Calcul du gain** :
```
gain_pp = (score_PhaseB - score_PhaseA) * 100
```

**Gate decision (ADR §3d)** :
- `gain ≥ 15pp` → **GO** — industrialiser les 5 tools (S3.5 complet, puis S3.6 compare_across_versions, puis S3.7 metrics)
- `gain < 15pp` → **NO-GO** — bloquer S3.5+ jusqu'à revoir S2.0 (bench Docling vs SmolDocling + extraction tables/numeric)

**Pourquoi cette borne** :
- 15pp = différence cohérente avec le gap mesuré (0.83 EKX − 0.57 V5 POC = 26pp). On vise à combler ~60% du gap quantitatif avec les tools seuls.
- Si <15pp, c'est que les tools opèrent sur des structures vides (V5Table=0 en Neo4j actuellement) — donc la cause racine est ailleurs.

## 4. Risques identifiés

| Risque | Probabilité | Mitigation |
|---|---|---|
| **Tables non extraites** : V5Table=0 dans Neo4j actuellement (Docling page-fallback ne fait pas l'extraction de tables) | **Élevée** | get_table retournera "no tables available" → mesure du gain SANS get_table, puis comparaison post-S2.0 |
| Panel 8q trop petit pour conclure → variance haute | Moyenne | Sanity check : si bench V5 sans tools donne score différent du bench S0 v2 (0.60 sur 3q quantitative), comprendre pourquoi |
| Charte domain-agnostic violée par implémentation pressée | Faible | grep -i "rto\|rpo\|sla\|sap" sur les handlers avant merge — auto-check via test pytest existant (`test_no_domain_specific_words_in_descriptions`) |
| Implémentation des 5 tools ≥ 3 jours | Élevée | Découper : prioriser find_quantitative + extract_numeric_evidence (2 tools les plus impactants). resolve_unit_or_alias + compute_derived_metric + get_table en second. |
