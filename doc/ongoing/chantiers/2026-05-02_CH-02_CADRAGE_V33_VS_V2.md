## 3. Ce que V2 retire de V3.3

| Concept V3.3 / Runtime V1.1 | Raison du retrait |
|---|---|
| Query Resolver 7 modes | Remplacé par Anchor Extractor (rôle plus simple) |
| 3 régimes RAG_LED/KG_LED/HYBRID | Pipeline linéaire sans branche conditionnelle |
| Auto-escalation RAG↔KG | Anchor Filter result-driven |
| kg_trust score pondéré par mode (7 facteurs) | Trop couplé aux modes — Trust score affiché en fin de pipeline simplement |
| Composabilité transitive 3 hops | Coût LLM élevé — déféré V3 explicite |
| Phase 4 Temporal Retrieval Layer (3 modes snapshot/weighted/diff) | Couvert par Anchor Filter range |
| Phase 5 DDL graph auditabilité formelle | R&D 30-60j — déféré |

| Response Composer 5-sections par mode | Réduit à réponse simple + drill-down |
| 3 personas (compliance_officer/explorer/reader) | Réduits à 1 toggle Audit |
| Détection CONFLICT runtime mode-driven | Remplacé par CONFLICT pré-calculé + remontée dans la réponse |

## 4. Ce que CH-02 doit livrer concrètement (proposition)

Le titre original "Sprint V3.3 contradiction detection" est trompeur car le runtime V1.1 a été déprisé. Le **vrai contenu actionnable** de CH-02 en V2 est :

### 4.1 Compléter la couverture lifecycle_status DocumentContext (gap critique)

- 0/17 docs ont `lifecycle_status` → bloque le Current Resolver
- Backfill via inférence runtime au demarrage (les docs ACTIVE sans LIFECYCLE_RELATION sortant les marque ACTIVE par défaut, ceux sortants d'un EVOLVES_FROM/SUPERSEDES → DEPRECATED si un successor existe)
- **Effort** : 0.5j (script Cypher déterministe)

### 4.2 Auditer la qualité des LOGICAL_RELATION existantes

- 4 319 EQUIVALENT (89%) — sur-classification probable, à investiguer
- 16 CONFLICT — vérifier qu'ils sont des vraies contradictions intra-anchor
- **Sortie** : un échantillon de 30-50 paires annotées humainement → décision : on garde / on purge / on re-classify
- **Effort** : 0.5-1j cadrage + 0.5j annotation user

### 4.3 Implémenter REAFFIRMS dans LIFECYCLE_RELATION Doc→Doc

- Actuellement seuls EVOLVES_FROM (3) et SUPERSEDES (1) sont implémentés
- REAFFIRMS = "doc B confirme doc A sans modification" (ex: `AMC 25.1419 reste applicable`)
- Pattern V2-S1 strict : déclaration textuelle explicite uniquement, validator evidence-locked
- **Effort** : 1j (réutilise le pattern V2-S1 existant)

### 4.4 Compléter validity_start claim-level (gap S1b non finalisé)

- 333 claims (0.83%) ont l'override, cible S1b était ~3 401 (8.5%)
- Le backfill Tier 4 LLM evidence-locked sur 3 401 candidates Tier 3 a été partiellement exécuté
- **Effort** : 1-1.5j (relancer le batch S1b sur EC2 vLLM, ~37 min compute)

### 4.5 Différer ce qui reste de V3.3 jusqu'à un V3 explicite

À ne **pas** entreprendre maintenant :
- Composabilité transitive 3 hops
- Temporal Retrieval Layer multi-modes
- DDL graph auditabilité formelle
- Re-classifier 12-types complet (le KG actuel est suffisant)
