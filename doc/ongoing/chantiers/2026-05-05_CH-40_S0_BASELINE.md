## 2. Contexte produit OSMOSIS

### 2.1 Positionnement
OSMOSIS construit un système RAG+KG pour Q&A sur corpus régulé. Différenciateur vs RAG vanilla : **graphe de claims atomiques (Neo4j) + chunks reconstruits (Qdrant) + détection automatique des relations cross-document** (contradictions, supersession lifecycle, qualifications).

Architecture conceptuelle (ADR_UNITE_PREUVE_VS_UNITE_LECTURE) :
- **Unité de preuve** = `Claim` Neo4j (médiane 113 chars), 40 196 instances aujourd'hui
- **Unité de lecture** = chunk Qdrant (médiane 1 401 chars), 21 630 instances
- 6 ancres documentaires par claim (verbatim_quote, passage_id, chunk_ids, page_no, unit_ids, content_fingerprint) → traçabilité native AI Act art. 12

### 2.2 Corpus actuel
- 17 documents PDF (~9 500 pages) : CS-25 amdt 22-28, EU 2021/821, EU 428/2009 (legacy abrogé), 4 actes délégués 2023-2024, 4 NPA/change_amdt, 4 ED Decisions
- 40 196 claims atomiques, 19 575 entités brutes → 2 320 entités canoniques (réduction 88%)
- 9 622 ClaimClusters, 68 facettes thématiques
- LOGICAL_RELATION cross-doc V3.3 typées : 4 289 (mostly EQUIVALENT, + 4 CONFLICT, 3 DISJOINT, 1 SUBSET, 1 DEFINITION_OF) + 2 965 CHAINS_TO

### 2.3 Pipeline V3 actuel (livré CH-39 le matin 2026-05-05)
5 stages, ~250 LoC (vs 951 V2). Domain-agnostic, zéro liste hardcoded.

```
QUESTION
   │
   ▼
[1] HYBRID RETRIEVE (Qdrant knowbase_chunks_v2)
    Dense (intfloat/multilingual-e5-large) + Sparse BM25 MatchText
    Fusion RRF (k=60), prefetch top_k×3
   │
   ▼
[2] CROSS-ENCODER RERANK GPU
    BAAI/bge-reranker-v2-m3 (568M, multilingue), GPU L4
    → top 10 final
   │
   ▼
[+] ENRICH METADATA (Neo4j)
    DocumentContext.lifecycle_status + publication_date
   │
   ▼
[3] AGENTIC SYNTHESIS — 1 SEUL APPEL LLM
    Politique RuntimeLLMClient : vLLM-first / DeepInfra-fallback.
    - Si Redis burst state actif (vLLM EC2 attaché) : Qwen2.5-14B-Instruct-AWQ (vLLM EC2 spot g6.xlarge L4)
    - Sinon (état Sprint S0 actuel) : Qwen2.5-72B-Instruct (DeepInfra)
    ⚠️ Tous les benchs S0 (V3_S0_BASELINE, V3_S0_GOLD4) ont tourné avec Qwen2.5-72B
       car vLLM EC2 inactif pendant la session. La projection « F1 60-75% sur Qwen-14B »
       qui apparaît dans certaines analyses est théorique sur le path vLLM-first à venir.
    Prompt unique 140 lignes avec décision ternaire ANSWER|REJECT_FALSE_PREMISE|ABSTAIN
   │
   ▼
[4] NLI FAITHFULNESS JUDGE
    MoritzLaurer/mDeBERTa-v3-base-xnli-multilingual-nli-2mil7 (278M, GPU L4)
    Split réponse en claims atomiques → softmax [ENTAIL/NEUTRAL/CONTRA] vs evidence
   │
   ▼
[5] REGEN CONDITIONNEL (1× max si UNFAITHFUL & score < 0.5)
```

## 4. Audits externes — 3 critiques sollicitées

3 LLMs experts indépendants (Claude web, ChatGPT, modèle d'analyse spécialisé) ont audité l'état V3 et la première proposition d'architecture V4. Convergences fortes :

### 4.1 Convergences (3/3 d'accord)
1. **Le problème est dans la synthèse + vérification, pas le retrieval** — ctx_relevance OK, faithfulness KO.
2. **mDeBERTa-xnli (278M, 2022) est obsolète** comme verifier production. État de l'art 2026 : MiniCheck-7B, Patronus Lynx-8B, Vectara HHEM-2.1.
3. **Le prompt 140 lignes / 11 règles est en surcharge cognitive** sur Qwen-14B.
4. **Le keyword fallback dans le scorer fausse les benchs** (mix judge + token-matching opaque).
5. **Aucun gold-set humain** pour calibrer le juge — juge non validé.
6. **RAGAS FactualCorrectness importable mais désactivé** par défaut.

### 4.2 Recommandations communes
1. **Question router** en amont (factual / list / temporal / causal / comparison / unanswerable)
2. **Adaptive retrieval** (top-K vs exhaustif doc-scoped vs multi-version)
3. **Verifier spécialisé grounding** (cascade : citation déterministe + NLI cheap + specialist)
4. **Régen ciblée phrase-level** (pas réponse entière)
5. **Gold-set humain 100-250q** stratifié multilingue
6. **Stack métriques hybride** : RAGAS FactualCorrectness + item-level recall + exact_match identifiers + citation rate

### 4.3 Critiques apportées à la 1ère synthèse V4 (corrections intégrées)
| Critique | Correction ADR |
|----------|----------------|
| **Gap multilingue MiniCheck/Lynx/HHEM** (English-trained) | D2 → cascade verifier multilingue, swap dogmatique remplacé par bake-off A/B/C |
| Evidence Structurer surdimensionné (extraction F1 60-75% sur 14B, +SPOF) | D3 → conditionnel sur hard routes uniquement, A/B avant déploiement, gate F1 ≥ 0.75 |
| Chiffres aspirationnels (-40% coût, p50 19s) | Reformulés en hypothèses H1-H5 à mesurer |
| Citation grounding ≠ support sémantique | Channel C1 = filtre minimal, pas proof of grounding |
| Sparse retrieval heuristique fait-main | D4 → Sprint S0.5 ajout : Qdrant native sparse / BGE-M3 |
| Routing 70/30 assumé | D6 → distribution mesurée sur gold-set |
| Decision Gate risque dérive V2 | D8 → max 5 règles, ablation +Xpp obligatoire par règle |
| « Convergence 3/3 LLM » est fragile | Hypothèses + bake-off + gold-set humain comme garde-fous |

### 4.4 Garde-fous anti-overfit Claude-juge
ChatGPT a soulevé : utiliser Claude comme reviewer du gold-set introduit 3 risques structurels.
1. Sur-optimisation au juge
2. Style over substance
3. Sous-pondération erreurs critiques

→ 3 mesures intégrées dans Sprint S0 :
- **Métriques structurées CRITIQUES** (CH-40.2) calculées sur 100% des samples — `exact_match`, `item_recall`, `citation_presence`
- **Logging désaccords judge vs structured** (CH-40.6) : top-20 cas overscored documentés
- **Sanity check externe Fred** (CH-40.7) : 9q ternaire OK/KO/bizarre

## 5. ADR Architecture V4 — 13 décisions

Source : `doc/ongoing/ADR_OSMOSIS_V4_ARCHITECTURE.md`

### Architecture cible (D1)
```
[A] Question Analyzer (multi-label top-2)
[B] Adaptive Retrieval (default | list | versioning | comparison)
[C] Evidence Structurer (CONDITIONNEL hard routes uniquement)
[D] Routed Synthesis (14B vLLM ou 72B DeepInfra selon type)
[E] Cascaded Multilingual Verifier (3 channels)
[F] Decision Gate (rule-based, max 5 règles)
```

### Décisions clés
| # | Décision | Justification |
|---|----------|---------------|
| D1 | Pipeline modulaire 6 stages | Modularité ComposeRAG, chemin court préservé pour 70% trafic |
| D2 | Verifier multilingue cascadé (mDeBERTa C2 + specialist C3 EN-only via bake-off) | MiniCheck/Lynx/HHEM majoritairement EN |
| D3 | Evidence Structurer **conditionnel** sur hard routes (temporal/contradiction/comparison/causal) | Latence + SPOF + F1 incertain |
| D4 | Sparse retrieval = Qdrant native ou BGE-M3 (pas heuristique) | `_extract_keywords_for_bm25()` brittle multilingue |
| D5 | Question Router multi-label top-2, **4 types** au démarrage (factual/list/temporal/causal_why) | Pas 6-8, multi-label gère non-disjoints |
| D6 | Routing modèle 14B/72B **appris** depuis gold-set, pas assumé 70/30 | Distribution réelle à mesurer |
| D7 | Régén **phrase-level** (pas réponse entière), 1× max | Économie latence -50% sur regen |
| D8 | Decision Gate rule-based, **max 5 règles**, ablation obligatoire | Anti-dérive V2 |
| D9 | Schéma sortie : `coverage_state` complete/partial/unknown + `exact_match_validation` first-class | Diagnostic admin UI |
| D10 | Suppression keyword fallback **complète** | Anti-pattern V2 |
| D11 | Gold-set humain **100-200q** stratifié multilingue, criterion-level | Calibration Pearson ≥ 0.7 obligatoire |
| D12 | Charte anti-V2 : max 50 lignes prompt, pas de listing métier hardcodé, max 5 règles gate | Vérifié en CI |
| D13 | Sprints à gates quantifiés go/no-go | Si gain < 3pp = no-merge |

### Hypothèses à valider (H1-H5)
- H1 : faithfulness vrai > 0.63 via FactualCorrectness → **réfuté** (factual 0.37)
- H2 : distribution trafic 70/30 simple/hard → **partiellement validé** (~70% factual+list+temporal sur le gold-set)
- H3 : MiniCheck-7B/HHEM-2.1 EN > mDeBERTa multilingue → à valider S1 bake-off
- H4 : prompt scindé < 50 lignes par route > prompt 140 lignes → à valider S2
- H5 : Evidence Structurer +5pp hard routes → à valider S3 A/B

**Interprétation détaillée** :
- factual / list / temporal : forte cohérence judge ↔ structured. Le gold-set capture bien la qualité.

→ Action S0 résiduelle (avant S1) : enrichir le schéma gold-set avec `correct_premise_rejection_signals` et `unanswerable_explicit_signals` (signaux dédiés multilingues domain-agnostic), re-mesurer Pearson.
