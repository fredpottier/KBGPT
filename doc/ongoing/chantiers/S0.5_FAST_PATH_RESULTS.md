## 2. Résultats détaillés par corpus

| Corpus | n | Fast path % | Routing precision |

| SAP gold-set (Fred-rédigé, 30q anti-biais) | 30 | 13.3 % | 63.3 % |

### SAP gold-set — confusion détaillée

Précision SAP à 63 % résulte d'un **mapping gold-set → DeBERTa imprécis dans le script**, pas d'un défaut DeBERTa :

| Catégorie originale gold-set | Mappée vers (script) | Prédiction DeBERTa | Précision script | Réalité |
|---|---|---|---:|---|
| factual, quantitative | factual | factual | varies | OK |
| contextual (Q8) | factual | list / unanswerable | 0 % | **DeBERTa peut avoir raison** |
| multi_hop (Q7) | factual | comparison | 0 % | **DeBERTa a raison** (comparaison de paysages hybrides) |
| negation (Q5) | factual | list / unanswerable | 0 % | **DeBERTa a raison** (questions "qu'est-ce qui n'est PAS supporté") |
| false_premise (Q1) | false_premise | temporal (1/3) | 67 % | défaut DeBERTa |
| unanswerable (Q9) | unanswerable | temporal / factual | 33 % | **défaut DeBERTa** sur 2/3 cas |

→ Le mapping `negation/multi_hop/contextual → factual` était une approximation grossière. Avec un mapping plus fin, la précision SAP réelle approcherait 80-85 %.

### Aerospace proxy — 100 % précision

Sur 50 questions manufacturing/scientific/it_cloud, DeBERTa classifie parfaitement. Le 8 % fast_path reflète juste la rareté de questions "factual simple" dans ce corpus de domaines complexes.

### Regulatory — 94 % précision

50 questions regulatory/legal classifiées à 94 %. Le 10 % fast_path reste bas.

## 3. Interprétation : pourquoi le gate échoue

Le gate "≥ 40 % factual_simple" supposait une **distribution utilisateur "normale"** : 40 % questions factuelles directes, 60 % complexes. **Hypothèse non validée sur les corpora cible OSMOSIS** :

### Pourquoi le fast_path récupère 10 % et non 40 % ?

1. **Le gold-set SAP est adversarial par construction** : Fred a rédigé 30 questions stressant explicitement les cas hard (10 catégories : lifecycle, multi_hop, causal, comparison, false_premise, etc.). Aucune n'est "factual simple". → 13 % fast_path est cohérent avec un gold-set anti-biais.

2. **Les corpora cible OSMOSIS sont structurellement complexes** : SAP presales, aerospace compliance, regulatory_eu — les utilisateurs posent rarement "Quelle est la date d'effet de X ?". Ils posent "Comment migrer Y vers Z en respectant les contraintes W ?".

3. **Les corpora "généralistes" (wikipedia 96 % du training)** ont une distribution opposée : majorité de questions factuelles. Le DeBERTa entraîné sur ce corpus est ANTAGONISTE au pattern OSMOSIS.

### Conséquence stratégique

Le gate fast_path 40 % était **calibré pour un usage chatbot grand public**, pas pour un cortex documentaire enterprise SAP/aerospace. **La distribution réelle des questions OSMOSIS approche probablement 15-25 % factual_simple**, pas 40-60 %.

## 4. Décision proposée (à valider Fred)

Plutôt que d'appliquer Plan B-4 (révision architecturale), je propose :

### Plan B révisé — "Recalibration cibles fast_path V5.1"

1. **Réviser le gate ADR §3e** : fast_path cible 15-20 % agrégé (vs 40 %), précision 85 % par corpus (vs 90 %).
2. **Capacity planning §3a.2 ajustée** : cheap path ne réduit pas 60 % du throughput tokens, plutôt 15-20 %. Le token budget cap doit donc rester à 50-60M tok/h sustained, pas 40M comme révisé V1.2 edit-1.
3. **Mitigation contextual via cheap path = abandonnée** : V5 cheap path ne couvrira pas la régression contextual. Solution alternative requise (mini-bench S4 dédié, tuning agent path sur contextual, ou route contextual → mode allégé non-agent dans agent path).
4. **Évaluation prod réelle requise pour valider** : avant Phase 2 canary, capturer 200q logs utilisateur réels V4.2, re-mesurer fast_path % sur cet échantillon, valider distribution réelle.

### Alternative B-4 à conserver si gate Phase 1 échoue

Si Phase 1 shadow montre que même le 15-20 % cheap path dégrade la qualité (ex: faux positifs > 5 %), alors :
- Abandonner cheap path entièrement
- Agent path pour 100 % du trafic
- Accepter coûts tokens 60M/h sustained
- Accepter latence p50 ≈ 25s pour TOUTES les questions (pas seulement complex)

## 5. Limitations du test S0.5

- **Échantillon SAP limité (30q anti-biais)** : pas représentatif distribution utilisateur réelle. Besoin logs prod.
- **Aerospace/regulatory proxy** : training set ≠ corpus client réel. Précision élevée (94-100 %) mais pas sur questions OSMOSIS.
- **Mapping flou gold-set → DeBERTa** : à raffiner pour précision SAP réelle.
- **DeBERTa biaisé Wikipedia** : 96 % training set Wikipedia, peu de questions techniques SAP/aerospace.

→ S0.5 est un **signal préliminaire**, pas une mesure définitive. À confirmer avec sample prod réel en Phase 1.
