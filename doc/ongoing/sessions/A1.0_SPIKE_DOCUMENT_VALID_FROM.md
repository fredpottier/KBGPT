## Implications pour A1.3

### Stratégie naïve ÉCARTÉE

La cascade actuelle du script spike (S2 keyword > S1 > S2 fallback > S3) est **inadéquate** car S1 metadata est globalement non fiable sur ce corpus.

### Nouvelle stratégie recommandée pour A1.3

**Cascade révisée** (priorité du + fiable au - fiable) :

1. **S2-keyword** — Texte page 1 proche d'un mot-clé sémantique ("Published", "Effective", "Version date", date du document)
   → Le plus fiable car contenu *officiel* du document
2. **S3-enrichi** — Nom de fichier avec patterns enrichis :
   - Année 4 chiffres `_2025_` → 2025-01-01
   - Mois + année EN/FR `_may_2025_`, `_june_2025_` → 2025-MM-01
   - Pattern compact `YYYYMMDD` (ex: `20251010`) → 2025-10-10
   - Version + année `_v04-2026_` → 2026-04-01
3. **S1-metadata** — Fallback de dernier recours, avec **warning systématique** dans les logs si la creation_date est récente (<6 mois après ingested_at, signal de re-save)
4. **S4 (nouveau)** — LLM extraction ciblée page 1 si S2/S3/S1 tous KO ou divergent fortement (>1 an d'écart). Reste evidence-locked (AX-3) : on extrait, on n'infère pas. Coût modeste : 1 call LLM/doc à l'ingestion, pas au runtime.

### Détection "batch re-save"

Ajouter une heuristique de défiance sur S1 :
- Si plusieurs docs partagent la même `/CreationDate` au jour près (≥3 docs) → S1 unreliable pour ce batch → escalade vers S2/S3/S4
- À implémenter en A1.3 comme pre-pass d'audit du corpus avant ingestion

### Champ `valid_from_source` (Phase A2) — confirmation utilité

L'ADR §2.2 prévoyait `marker_type` Phase A2 pour distinguer `explicit | document_inherited | ingestion_fallback`. Le spike **confirme** que ce champ est critique : sur ce corpus, ~60% des claims auraient `valid_from = ingested_at` (fallback) si on appliquait la stratégie naïve, et la classification temporelle Phase A2 doit pouvoir filtrer ces claims-là.
