## 2. Détail par test

### 2.1 T1 — Coverage `ingested_at`

**Critère** (révisé §9.8) : 100 % des claims du tenant ont `ingested_at` non-NULL. `valid_from = NULL` est légitime depuis la décision S1 désactivé (§9.1) et n'est plus compté comme manquant.

**Résultat** :
- Total claims : 14 417
- Avec `ingested_at` : 14 417 (100 %)
- Missing `ingested_at` : **0** ✅

**Note** : tous ces claims sont issus de la migration A1.2 (qui a peuplé `ingested_at = created_at` pour les claims pré-existants) ou de la voie ClaimFirst A1.3 (qui pose `ingested_at = datetime.now(UTC)`). Aucun claim sans `ingested_at` n'a été observé.

### 2.2 T2 — Timestamps du doc le plus récent

**Critère** : sur le doc le plus récemment ingéré, tous les claims associés ont `ingested_at` non-NULL et les autres timestamps sont cohérents (NULL acceptable pour `valid_from`, `valid_until`, `invalidated_at`).

**Résultat** (doc proxy = `007_SAP_S4HANA_PCE_-_Access_to_Customer_Systems_6ac19896`) :
- Total claims du doc : 58
- Missing `ingested_at` : **0** ✅
- Avec `valid_from` : 58 (100 %)
- Avec `valid_until` : 0 (normal, pas d'invalidation A2 encore)
- Avec `invalidated_at` : 0 (normal, pas d'invalidation A2 encore)

**Limite probative** : ce doc est issu de la migration A1.2 (`valid_from = ingested_at` par défaut), pas du flux A1.3 actuel qui produit potentiellement `valid_from = NULL` quand S1 est désactivé et S2/S3/S4 échouent. Le test garantit l'invariant fort (`ingested_at` obligatoire) mais ne valide pas encore le flux NULL légitime. **Confirmation visuelle attendue** lors de la réingestion à venir (#314 PHASE A2 — 38 docs).

### 2.3 T3 — Queries point-in-time §2.4

**Critère** : les 3 queries de référence de l'ADR §2.4 (now / as-of past / invalidated-since past) retournent des résultats cohérents.

**Résultat** :
- Échantillon stratifié : 9 claims répartis sur 5 docs distincts (2 par doc max)
- **Q1** "Quoi est vrai aujourd'hui ?" → 14 417 (tous actifs, attendu)
- **Q2** "Quoi était vrai au 2024-01-15 ?" → 0 (tous les claims ont été ingérés après 2024-01-15, donc filtre `ingested_at <= 2024-01-15` rejette tout — cohérent avec la timeline du projet)
- **Q3** "Qu'a-t-on invalidé depuis le 2024-01-15 ?" → 0 (aucune invalidation A2 n'existe encore — attendu)

Les 3 queries s'exécutent sans erreur. La cohérence sémantique est validée : Q3=0 traduit le fait qu'A2 n'est pas encore implémentée, pas un bug de la query.

**Note §9.8** : le filtre `valid_from` accepte désormais NULL — implémenté dans le script `(c.valid_from IS NULL OR c.valid_from <= datetime())`. Pour le KG actuel (post-migration A1.2 où `valid_from` est toujours non-NULL), ce filtre est équivalent à l'ancien. Il sera utile dès que la réingestion produira des `valid_from = NULL`.

### 2.4 T4 — Indexes & EXPLAIN

**Critère** : les 3 indexes `claim_active`, `claim_event_time`, `claim_ingested` sont créés et ONLINE. Le plan EXPLAIN d'une query type utilise un de ces indexes (preuve d'efficacité).

**Résultat** :
- Indexes attendus : 3
- Indexes trouvés : 3 (tous **ONLINE**)
- Missing : aucun
- Plan EXPLAIN sur Q1 : contient un opérateur `NodeIndexSeek` (ou équivalent) → `plan_uses_index = True` ✅

### 2.5 T5 — Perf p50 / p95 / p99

**Critère** (ADR §3.3 test 5, bloquant) :
- p50 < 100 ms
- **p95 < 500 ms** (bloquant)
- p99 < 1 s

**Résultat** sur 20 répétitions × 3 queries = 60 mesures agrégées :

| Percentile | Mesure | Seuil | Marge |
|---|---|---|---|
| p50 | **2.5 ms** | 100 ms | × 40 sous le seuil |
| **p95** | **20.6 ms** | **500 ms** | × 24 sous le seuil bloquant |
| p99 | 21.4 ms | 1 000 ms | × 47 sous le seuil |
| max | (observé via per-query) | — | — |

**Conclusion** : les indexes composites `(tenant_id, ...)` rendent les queries point-in-time triviales en termes de latence sur le KG actuel (14 417 claims). Aucune optimisation supplémentaire nécessaire à ce stade. Performance scalable raisonnablement à plusieurs centaines de milliers de claims (à re-mesurer après réingestion massive en A2).
